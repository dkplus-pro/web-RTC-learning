# 第 04 课：ICE 候选与本地信令模拟

本课在上一课的基础上加入 ICE 候选交换。你会看到浏览器如何收集候选地址、如何把候选发送给对端，以及为什么生产环境通常需要 STUN/TURN 服务提升连通率。

## 学习目标

- 理解 ICE 的职责：在多种网络路径中寻找可以互通的一条。
- 区分 SDP 协商和 ICE 候选交换：前者描述会话能力，后者帮助建立传输路径。
- 观察 `icecandidate`、`iceconnectionstatechange` 与 `connectionstatechange` 的关系。
- 认识本地信令模拟器的边界：它展示消息顺序，不代表生产信令可靠性。

## 关键概念

### ICE Candidate 是可尝试的网络路径

候选可能来自本机网卡、路由器映射地址或 TURN 中继。浏览器会把候选组合成候选对并进行连通性检查，最终选择可用路径。

### 信令不是 WebRTC 标准的一部分

WebRTC 定义了媒体、数据和传输协商 API，但没有规定信令服务器协议。工程团队需要根据鉴权、房间模型、重连策略和可观测性设计自己的信令层。

## 运行方式

```bash
npm run lesson:04
# 浏览器打开 http://localhost:5104
```

## 实验步骤

1. 点击“启动本地媒体”。
2. 点击“创建会话”，观察 offer/answer 消息进入本地信令队列。
3. 点击“交换 ICE 候选”，观察候选数量、连接状态和远端视频。
4. 多次重置实验，比较候选收集完成前后建立连接的差异。

## 架构提醒

生产信令需要处理消息乱序、重复投递、断线重连、房间成员变更和权限校验。ICE 失败时，应给用户可理解的反馈，并在服务端记录网络环境、候选类型和 TURN 使用情况。
