# 第 08 课：生产架构：信令、STUN/TURN、安全与演进

最后一课把前面学到的浏览器 API 放进生产系统架构中。WebRTC 本身只负责媒体和数据通道，业务仍然需要用户身份、房间状态、信令、NAT 穿透、TURN 成本控制、可观测性、录制、审核和安全策略。

## 你将学到

- 信令服务在 offer、answer、ICE candidate 交换中的职责边界。
- STUN 与 TURN 的区别，以及何时必须接受 TURN 中继成本。
- 生产系统的安全基线：HTTPS、鉴权、房间权限、设备权限提示、日志脱敏。
- 如何分阶段演进：单房间原型 → 多人会议 → SFU/录制/质量监控。

## 运行方式

```bash
cd lessons/08-production-architecture
python3 -m http.server 5108
# 打开 http://localhost:5108
```

这个 demo 不连接真实服务器，而是一个可交互的架构推演器。点击阶段按钮后，页面会展示对应系统组件、关键风险和架构决策，适合在团队设计评审时使用。

## 架构提示

当用户数量超过两人，或者需要录制、旁路直播、服务端混流、内容审核时，通常要引入 SFU。TURN 不是 SFU：TURN 只中继网络包，SFU 理解 RTP/RTCP 并做订阅转发。两者经常同时存在，但解决的问题完全不同。
